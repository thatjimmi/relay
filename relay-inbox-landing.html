Why is it called
Relay?

relay /rɪˈleɪ/ — verb
To receive and pass on information or a message. A relay doesn't originate — it faithfully carries what arrives to where it needs to go, without dropping it.
relay race — noun
A race in which each runner carries a baton and hands it off to the next. The baton is never dropped. Relay.Inbox is the baton handoff between an external system and your business logic.
relay station — noun
An intermediate station that receives a signal and retransmits it. The inbox is that station — it catches everything that arrives, holds it reliably, and lets your processor pick it up when ready.
electrical relay — noun
A switch activated by a signal in one circuit to control another circuit. Relay.Inbox decouples reception from processing — the signal (message arrival) triggers storage, independently of when your logic runs.

Features & how they work

🔑
Key derivation from data
Your handler declares how to compute the idempotency key from the message itself. The same trade arriving 50 times produces the same key — stored once, processed once. The caller never has to think about it.
IInboxHandler<T>
🏷️
Named, isolated inboxes
Register separate inboxes for separate concerns — "market-exchange", "payments", "notifications". Each is an isolated channel with its own handlers, options, and query scope. Zero cross-contamination.
AddInbox("name")
📥
Safe receive
IInboxReceiver<T> is the single entry point at your boundary. It derives the key, checks the unique DB index atomically, and stores the message. Returns a result — never throws for duplicates. Tag each message with its source feed.
IInboxReceiver<T>
⚙️
You control processing
No hidden background threads. Call IInboxProcessor.ProcessPendingAsync() from a Hangfire job, Azure Function timer, minimal API endpoint, or a test. You decide the cadence, the batch size, and the trigger.
IInboxProcessor
🔄
Retry & dead-lettering
Failed messages are retried up to MaxRetries, with the retry count persisted in the DB. After exhausting retries the message is dead-lettered and the OnDeadLettered hook fires — perfect for alerting on-call or pushing to a queue.
MaxRetries · OnDeadLettered
🔒
SKIP LOCKED — parallel safe
The SQL fetch uses WITH (UPDLOCK, READPAST). Multiple processor instances running in parallel each claim a distinct batch at the DB level. No distributed locks. No coordination. Just scale out.
UseSkipLocked
📊
Inbox stats & health
GetStatsAsync() returns pending, processing, processed, failed, and dead-lettered counts — plus the timestamp of the oldest pending message. Wire this directly into a health check endpoint or dashboard.
IInboxQuery
♻️
Requeue dead letters
Requeue a single dead-lettered message by ID, or bulk-requeue all of them with one call. Retry count resets to zero. Useful during incidents when a downstream dependency was temporarily unavailable.
IInboxRequeue
🧹
Purge old messages
PurgeProcessedAsync() deletes successfully processed messages older than a given timespan. Run nightly to keep the table lean. Schema and indexes stay tight — no bloat creep over time.
PurgeProcessedAsync
🔗
Distributed trace correlation
TraceId is captured automatically from Activity.Current when a message is received. Your stored inbox rows are correlated with your distributed traces without any manual wiring — search by trace ID in your APM.
TraceId · Activity.Current
🪝
Lifecycle hooks
OnProcessed, OnFailed, OnDeadLettered, OnDuplicate — async hooks on every state transition. Emit custom metrics, write audit logs, trigger alerts, or push to a secondary queue. Zero coupling to internals.
InboxOptions
🔌
Swappable storage
IInboxStore is the only interface the engine cares about. Ships with SqlInboxStore (raw ADO.NET, no ORM) and InMemoryInboxStore. Bring your own Postgres, MongoDB, EF Core, or Redis implementation.
IInboxStore

Step by step

1

Message arrives at your boundary

A WebSocket frame, a webhook POST, a Kafka message. You deserialize it to a typed record and call IInboxReceiver<T>.ReceiveAsync(msg). That's your entire entry point.

2

Key is derived from the message data

The handler's GetIdempotencyKey(msg) runs. The result is scoped to the inbox name — e.g. market-exchange:trade:NYSE:TRD-9981. No caller input required.

3

Duplicate check at the DB level

A unique index on IdempotencyKey makes the duplicate check atomic and race-condition safe. Even if two threads arrive simultaneously with the same message, only one insert wins. The other gets a clean duplicate result.

4

Message stored with full metadata

Status, source tag, TraceId from the active trace context, and ReceivedAt timestamp all written in a single insert. The message is now durable — a crash here loses nothing.

5

You trigger processing when ready

Call IInboxProcessor.ProcessPendingAsync("inbox-name") from wherever fits. Batch size is configurable. SKIP LOCKED means multiple instances can run safely in parallel.

6

Handler runs, lifecycle hooks fire

Message is deserialized back to the typed record and dispatched to your IInboxHandler<T>.HandleAsync(). On success, OnProcessed fires. On failure, retry count increments. On exhaustion, OnDeadLettered fires.

In practice

// Your handler owns two things:
// 1. How to derive the key from data
// 2. What to do with the message

public record TradeExecutedEvent(
    string Exchange,
    string ExternalTradeId,
    string Symbol,
    decimal Price,
    decimal Quantity);

public class TradeExecutedHandler : IInboxHandler<TradeExecutedEvent>
{
    // Same trade from same exchange = same key, always.
    // Arriving 50 times = processed exactly once.
    public string GetIdempotencyKey(TradeExecutedEvent msg) =>
        $"trade:{msg.Exchange}:{msg.ExternalTradeId}";

    public async Task HandleAsync(TradeExecutedEvent msg, CancellationToken ct)
    {
        // your business logic — runs exactly once per unique trade
        await _portfolio.RecordTradeAsync(msg);
    }
}
builder.Services
    .AddInbox("market-exchange", inbox => inbox
        .WithHandler<TradeExecutedEvent, TradeExecutedHandler>()
        .WithHandler<OrderFilledEvent, OrderFilledHandler>()
        .WithMaxRetries(5)
        .OnDeadLettered(async (msg, ex) =>
            await alerts.NotifyAsync($"Dead: {msg.Type} {msg.Id}"))
        .OnProcessed(msg =>
            metrics.Increment("inbox.processed")))

    .AddInbox("payments", inbox => inbox
        .WithHandler<PaymentReceivedEvent, PaymentReceivedHandler>())

    .UseSqlInboxStore(connectionString, opts =>
    {
        opts.UseSkipLocked = true;   // parallel-safe processing
        opts.AutoMigrateSchema = true; // create table on startup
    });
// At your WebSocket/webhook/consumer boundary
public class MarketFeedListener(IInboxReceiver<TradeExecutedEvent> inbox)
{
    public async Task OnTradeAsync(TradeExecutedEvent trade)
    {
        var result = await inbox.ReceiveAsync(trade, source: "binance-ws");

        if (result.WasDuplicate)
            _log.LogDebug("Duplicate ignored: {Id}", trade.ExternalTradeId);

        // That's it. Message is safe in the DB.
        // No need to process it here — decouple receive from handle.
    }
}
// Option A: Minimal API endpoint (trigger via cron)
app.MapPost("/inbox/process/{inbox}",
    async (string inbox, IInboxProcessor processor) =>
        Results.Ok(await processor.ProcessPendingAsync(inbox, batchSize: 100)));

// Option B: Hangfire recurring job
RecurringJob.AddOrUpdate<IInboxProcessor>(
    "market-inbox",
    p => p.ProcessPendingAsync("market-exchange", 100, CancellationToken.None),
    Cron.Minutely());

// Inspect + manage
var stats   = await store.GetStatsAsync("market-exchange");
var dead    = await store.GetDeadLetteredAsync("market-exchange");
await store.RequeueAllDeadLetteredAsync("market-exchange");
await store.PurgeProcessedAsync("market-exchange", TimeSpan.FromDays(30));
// Zero infrastructure needed — InMemoryInboxStore ships in the package
[Fact]
public async Task Trade_is_processed_exactly_once()
{
    var store    = new InMemoryInboxStore();
    var handler  = new FakeInboxHandler<TradeExecutedEvent>(
        t => $"trade:{t.Exchange}:{t.ExternalTradeId}");
    var receiver = new InboxReceiver<TradeExecutedEvent>(store, handler, "market", new());

    var trade = new TradeExecutedEvent("NYSE", "TRD-001", "AAPL", 189.50m, 100);

    await receiver.ReceiveAsync(trade);
    await receiver.ReceiveAsync(trade); // duplicate — silently dropped

    Assert.Single(handler.Handled); // handled exactly once
    Assert.Equal(1, store.All.Count); // stored exactly once
}

Why use Relay?

Most teams either skip the inbox pattern entirely and lose messages, or bolt on a heavy framework that brings more complexity than the problem it solves. Relay is the middle path.

Without Relay
With Relay
Process messages inline — if your handler throws, the message is gone forever
Messages hit the DB first. Handler failure means retry, not loss.
Hand-roll idempotency checks in every handler, different each time
Key derivation lives on the handler contract — impossible to forget.
Duplicate messages from flaky sources cause double-processing
DB-level unique index catches duplicates even under race conditions.
Pull in MassTransit or NServiceBus — hundreds of concepts to learn
Four interfaces. One NuGet package. Zero broker required.
Tests require a real database or complex mocking
InMemoryInboxStore ships in the package — test with zero infrastructure.
Scale out processors = distributed lock nightmares
SKIP LOCKED makes parallel processors safe by default.
Dead messages disappear silently — you find out from a customer
Dead-letter hook fires immediately. Requeue with one call.
